# Risk Register — Malaika Honey FRM

Likelihood/Impact rated Low/Medium/High. Review this list at the start of each new milestone (see [[Backlog]]) and after any incident.

| ID | Risk | Likelihood | Impact | Mitigation | Owner |
|---|---|---|---|---|---|
| R1 | ~~Open Firestore Security Rules (no auth in v1) allow anyone with the project config to read/write farmer and purchase data if the config leaks or is guessed~~ | ~~Medium~~ | ~~High~~ | ✅ Addressed — Firebase Auth (Google Sign-In) added; every `farmers`/`purchases`/`devices` rule requires both `request.auth != null` **and** the signed-in email existing in the `allowedStaff` allowlist (see [[Database-Schema]] and `firestore.rules`) | Dev lead |
| R16 | Sign-in is self-service (anyone can create a phone+password account, or — if `GOOGLE_SIGNIN_ENABLED` is ever re-enabled — attempt Google Sign-In) — the `allowedStaff` allowlist is the only thing standing between "anyone who can type a phone number" and full farmer/purchase data access | Low | High | Every data-touching Firestore rule requires `exists()` against `allowedStaff/{email}`, not just `request.auth != null` — an unapproved sign-in gets a Firebase Auth session but zero data access, surfaced client-side as an "Approval Needed" screen. This depends entirely on the rules file staying correct; treat any change to `firestore.rules` around this check as high-risk and test it explicitly before deploying (see [[QA-Testing]]) | Dev lead |
| R17 | Revoking a staff member's access (deleting their `allowedStaff` entry) doesn't take effect on a device that already has a cached "authorized" flag until that device next reaches the server | Low | Medium | Accepted, inherent to offline-first design — the same lag already exists for the sync queue generally (see R4). If someone needs to be blocked urgently, also disable their account in Firebase Console → Authentication (Users → disable), which *is* enforced immediately server-side regardless of any client-side cache | Ops manager |
| R18 | ~~`authDomain` in `firebase.config.js` didn't match the app's actual Firebase Hosting domain (`.firebaseapp.com` vs. `.web.app`) — this made Google Sign-In's redirect/popup round trip cross-origin, which Safari's cross-site tracking prevention silently blocked, bouncing users back to the login screen with no visible error on iOS in particular~~ | ~~Medium~~ | ~~High~~ | ✅ Addressed — `authDomain` now matches the actual hosting domain (see [[Config-Management]] "Firebase Web App configuration"). This class of bug is easy to reintroduce if the hosting domain ever changes (e.g. a future custom domain) without updating `authDomain` to match — re-test Google Sign-In on an actual iOS Safari device (not just desktop Chrome) after any hosting-domain change | Dev lead |
| R19 | A misconfigured `newFarmerFields` document (invalid `type`, a dangling `optionsSource` pointing at nothing, or every field accidentally deactivated) breaks the New Farmer form for every device that syncs the change — there's no schema validation on the admin-editing side (Firebase Console just accepts any document shape) | Medium | Medium | Test any `newFarmerFields`/reference-data edit against the local emulator first (see [[QA-Testing]]) before making the same change in production Console. Full Name and Phone are hardcoded outside this schema specifically so the form can never be fully broken (a farmer can always be minimally identified even if every schema-driven field is misconfigured) | Dev lead |
| R20 | Reference-data collections (`products`, `grades`, `paymentMethods`, `farmSizes`, `districts`, `newFarmerFields`) ship empty and only use the hardcoded fallback while completely empty — adding even one document to a previously-empty collection silently drops every other hardcoded default (not a merge). An admin adding "just the new option" without also re-adding the existing ones would unintentionally remove them from the form | Medium | Low | Documented explicitly in [[Config-Management]] "Editing reference data" with the exact failure mode spelled out; the fix if it happens is simply re-adding the missing documents, no data is lost (farmers/purchases already saved keep whatever value they had) | Dev lead |
| R21 | An admin account (`allowedStaff.role == 'admin'`) can approve *any* Google account it sees a `signupRequests` entry for, directly from the app — if an admin's own Google account were compromised, the attacker could approve their own or an accomplice's account for full farmer/purchase data access | Low | High | The blast radius is bounded: an admin can only ever grant plain staff access via the app (never the `admin` role itself, which stays Console-only by design — see [[Database-Schema]] "Staff accounts"), so a compromised admin account can't self-escalate further or create additional admins. Treat admin Google accounts with the same care as the Firebase project owner login; enable 2-step verification on them | Management |
| R22 | Rejecting a `signupRequests` entry doesn't block the person from signing in and generating a fresh `pending` request later — there's no permanent-deny state | Low | Low | Deliberate: a hard permanent-block with no unblock path seemed harsher than warranted for a mis-click or a since-resolved situation (e.g. someone who left and rejoined). If abuse ever becomes a real problem, the fix is straightforward — treat a `rejected` status as terminal in `recordSignupRequest` instead of resetting it to `pending` | Dev lead |
| R23 | A phone number is a weaker identity signal than a Google account — creating a phone+password account (`createAccountWithPhone`) doesn't verify the phone number belongs to the person typing it (no SMS/OTP step); anyone could type someone else's number | Low | Medium | The actual access boundary is unaffected — real data access still requires separate admin approval in `allowedStaff` regardless of which sign-in method reached the "Approval Needed" screen (see [[Database-Schema]] "Staff accounts"), so this only affects *whose name* shows up requesting approval, not whether unapproved access is possible. Admins should confirm a request's name/phone matches a real known staff member (e.g. by asking them directly) before approving, the same diligence already expected for Google-based requests | Ops manager |
| R24 | Office+code sign-in (the primary method as of v0.6.0) means every purchase/registration recorded from a given office shows the *office* in `recordedBy`/`registeredBy`, not which individual staff member actually entered it — a real loss of per-person accountability compared to the phone/Google methods it replaced | Medium | Low | Deliberate trade-off the client chose for simplicity given staff literacy constraints. If per-person accountability is ever needed again (e.g. a specific dispute), it would need a lightweight addition on top - such as prompting for a name once per session purely for the record fields - without changing how sign-in itself works | Management |
| R25 | Office codes are shared secrets - every staff member at an office uses the same one. If one person who knew it leaves, there's no way to revoke just their knowledge of it; the only fix is rotating the whole office's code (Console: reset that Auth account's password), which then requires re-informing everyone still at that office | Medium | Low | Accepted, inherent to a shared-credential design. Encourage rotating an office's code periodically and immediately after any departure, and document the current code somewhere the whole office can reach, not just the person who left | Ops manager |
| R26 | `fieldOffices` is the one collection in the whole schema readable with **no sign-in at all** (`allow read: if true` — needed so the Login screen's office picker works before anyone is authenticated) | Low | Low | Scoped deliberately narrow: the collection only ever contains office display names and ordering, nothing sensitive. Any future field added to this collection should be treated as public information by default, unlike every other collection in the schema | Dev lead |
| R2 | Field device lost/stolen with offline-queued, unsynced purchase or registration data | Medium | Medium | Firestore's offline queue is local to the device; encourage staff to sync (bring device online) before end of day; the login session persists per device, so a stolen device still exposes queued data until the account is disabled in Firebase Console — treat lost-device reporting as urgent | Ops manager |
| R3 | Two staff register a farmer or record a purchase simultaneously while both offline, then reconnect — risk of duplicate FRN or conflicting writes | Low | Medium | **Rewritten 2026-07:** the original mitigation ("a Firestore transaction resolves safely on sync") was incorrect — transactions fail immediately when offline instead of queuing, which is why they were removed. FRNs are now minted entirely client-side from a per-device code + local sequence (`MH` + deviceCode + sequence, e.g. `MHA1000042`), so two devices can never collide by construction — no server round-trip or coordination needed, online or off. Purchases remain additive (new documents) and don't conflict. See [[Database-Schema]] `devices/{deviceCode}` | Dev lead |
| R4 | Poor/no mobile network at buying centres prevents timely sync, delaying visibility of purchases to management | High | Low | This is the expected normal case, not a failure — the app is designed offline-first; management reporting should assume a sync lag, not real-time data ([[System-Architecture]]) | Ops manager |
| R5 | Staff mis-key weight, price, or grade with no edit/void flow in v1 | Medium | Medium | Emphasize double-checking the total before saving in [[Admin-User-Manual]]; add edit/void with audit trail in [[Backlog]] 2.6 | Dev lead |
| R6 | Sensitive personal data (farmer name, phone, location, income) held without explicit data-protection/consent language, ahead of any NGO data-sharing | Medium | High | Registration agreement (Section D of the paper form) should be extended to cover data use for M&E/reporting purposes before that data is shared externally; consult Uganda's Data Protection and Privacy Act 2019 requirements before any external data export | Management |
| R7 | Low-end Android phones / weak connections make the app feel slow or fail to load if it grows beyond a lightweight static app | Low | Medium | Deliberately avoided frameworks/build tooling; kept bundle minimal (see [[System-Architecture]]); test on real low-end hardware before each release ([[Release-Management]]) | Dev lead |
| R8 | Firebase free-tier quota (Spark plan) exceeded as farmer/purchase volume grows, causing writes to fail | Low | Medium | Monitor usage in Firebase Console; budget for Blaze (pay-as-you-go) plan before quotas are likely to be hit; Blaze still has a generous free allotment | Ops manager |
| R9 | Sole reliance on one developer's knowledge of the system (bus factor) | Medium | High | This documentation set ([[System-Architecture]], [[Database-Schema]], [[Config-Management]]) exists specifically to reduce this; keep it updated as the app changes | Management |
| R10 | No automated backups of Firestore data | Low | High | Schedule periodic manual exports (Console → Firestore → Import/Export to Cloud Storage) until volume/importance justifies automating; treat data loss as a business-continuity risk given this is the farmer payment record of truth | Ops manager |
| R11 | Currency assumption (UGX only, no decimals) breaks if Malaika expands to export pricing in USD or another currency | Low | Low | Currency handling is centralized in `public/js/lib/constants.js` / formatting utils, not scattered — a currency field can be added without a full rewrite | Dev lead |
| R12 | Printable farmer ID card contains personal data (name, FRN, phone) and could be lost/copied, enabling fraud (someone else presenting a farmer's card) | Low | Medium | Buying centre staff should visually confirm the person matches registration details on file (and, once added, the photo — [[Backlog]] 2.5) before processing a purchase against a presented card | Ops manager |
| R13 | Phone-number uniqueness on registration is a check-then-act client query, not a database constraint — two staff registering the same phone number at the exact same instant on two different offline devices could both pass the check before either syncs | Low | Low | Accepted for now given how rarely this timing window would occur in practice; see [[Database-Schema]] "Phone uniqueness" for the technical detail and the real-constraint alternatives if this ever needs hardening | Dev lead |
| R14 | Staff signs out (or the app force-navigates to `/login` on session expiry) while a purchase/registration write is still queued and unsynced | Low | Medium | The logout button checks the live sync badge state and shows an explicit warning ("hasn't finished syncing yet... sign out anyway?") before proceeding if anything is pending, rather than silently discarding the risk; staff are trained to wait for "Synced" before signing out when possible ([[Admin-User-Manual]]) | Dev lead |
| R15 | Reconciling an unverified purchase (`/reconcile`) lets staff silently correct the FRN on a saved purchase to any farmer that matches their search, with no second confirmation step or audit note beyond `frnUnverified` flipping to `false` | Low | Medium | Accepted for the MVP given reconciliation is staff-initiated and search-based (not a blind auto-match); if mis-linking becomes a real problem in practice, add a confirmation step and/or an audit trail of what the original typed FRN was vs. what it was resolved to — `originalTypedFrn` is already preserved on the purchase document, so this is retrievable even without a dedicated audit log | Dev lead |

## Review cadence

Revisit this register at minimum every milestone boundary in [[Backlog]], and immediately after any security incident, data loss event, or before agreeing to share data externally with an NGO/development partner.
